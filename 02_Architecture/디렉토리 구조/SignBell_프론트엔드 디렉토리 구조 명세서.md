# SignBell 프론트엔드 디렉토리 구조

* **QA / 테스트 엔지니어**: UI/UX 테스트 및 기능 테스트 설계 시 구조 참고
* **신규 합류자**: 프론트엔드 코드 베이스의 전체적인 구조를 빠르게 파악


## 2. 최상위 디렉토리 구조

```

frontend/
├── .github/  \# GitHub 관련 설정 (워크플로우, 템플릿)
├── public/   \# 정적 파일 (빌드 시 루트 경로에 복사됨)
└── src/      \# 소스 코드 루트 디렉토리

```

* `.github/`: GitHub Actions 워크플로우, 이슈/PR 템플릿 등 GitHub 관련 설정 파일 위치.
* `public/`: 빌드 과정에 포함되지 않고 그대로 복사되는 정적 파일 (예: `index.html`, 외부 라이브러리 파일) 위치.
* `src/`: React 애플리케이션의 모든 소스 코드(컴포넌트, 페이지, 스타일 등)가 위치하는 최상위 디렉토리.

## 3. `src` 디렉토리 구조

React 애플리케이션의 핵심 소스 코드가 위치합니다.

```

frontend/src/
├── assets/       \# 이미지, 폰트 등 정적 에셋
├── components/   \# 재사용 가능한 UI 컴포넌트
├── contexts/     \# React Context API 관련 파일
├── data/         \# 정적 데이터 (약관 등)
├── hooks/        \# 커스텀 훅 (재사용 가능한 로직)
├── layout/       \# 페이지 레이아웃 컴포넌트
├── pages/        \# 페이지 단위 컴포넌트
├── routes/       \# 라우팅 설정
├── services/     \# API 호출 및 외부 연동 로직
├── store/        \# 전역 상태 관리 (Zustand)
├── styles/       \# 전역 스타일 및 SCSS 변수
├── test/         \# 테스트 관련 파일 (주로 WebRTC 테스트용)
└── utils/        \# 범용 유틸리티 함수

```

* `assets/`: 이미지, 폰트 파일 등 프로젝트에서 사용되는 정적 자원 관리.
* `components/`: 버튼, 모달, 카드 등 여러 페이지나 다른 컴포넌트에서 재사용될 수 있는 작은 UI 단위. 기능별(auth, common, quiz 등)로 하위 폴더 구성.
* `contexts/`: 전역적으로 상태나 함수를 공유하기 위한 React Context API 관련 코드 위치 (예: Janus WebRTC 설정).
* `data/`: 코드 내에서 직접 사용하는 정적인 데이터 (예: 약관 텍스트) 관리.
* `hooks/`: 상태 관련 로직, API 호출 로직 등 반복적으로 사용되는 로직을 분리한 커스텀 훅 함수들 위치.
* `layout/`: 헤더, 푸터, 사이드바 등 페이지들의 공통적인 구조를 정의하는 레이아웃 컴포넌트 위치.
* `pages/`: 특정 라우트 경로에 매핑되는 페이지 단위의 컴포넌트. 각 페이지는 여러 `components`를 조합하여 구성. 기능별로 하위 폴더 구성.
* `routes/`: 애플리케이션의 전체 페이지 라우팅 구조를 설정하는 파일 위치.
* `services/`: 백엔드 API 호출, WebSocket 통신 등 외부 서비스와의 연동 로직을 담당하는 함수/클래스 위치.
* `store/`: Zustand를 사용한 전역 상태 관리 로직 위치. 인증 상태, 웹캠 상태 등을 관리.
* `styles/`: 전역 CSS/SCSS 파일, 공통 변수, 믹스인 등 스타일 관련 파일 위치.
* `test/`: 개발 중 특정 기능(주로 WebRTC)을 테스트하기 위한 임시 파일이나 스크립트 위치.
* `utils/`: 특정 도메인에 종속되지 않는 범용적인 유틸리티 함수들 위치.
