
## 트러블슈팅: 쿠키 만료 후 클라이언트의 로그아웃 지연 문제

* **작성자:** [송민재](https://github.com/songkey06)

-----

### 1\. 문제 현상 (Problem)

> 서버의 인증 토큰(쿠키)이 만료된 후에도 클라이언트(프론트엔드)가 로그인 상태를 유지하여 로그아웃 처리가 지연됩니다.

* **문제 1**: 브라우저 쿠키의 Access/Refresh Token이 만료되었으나, 프론트엔드가 로컬 스토리지의 `isLogin: true` 같은 플래그에 의존하여 로그인 상태를 유지합니다.
* **문제 2**: 토큰 만료 후, 로그인 상태에서만 접근 가능한 페이지 이동 시 자동으로 로그아웃이 발생하지 않습니다.
* **문제 3**: API 요청 시 서버에서 401(Unauthorized) 응답을 반환하지만, 페이지를 새로고침(F5)할 때 비로소 로그아웃 처리가 됩니다.

<br>

**[문제 상황 스크린샷 및 로그]**

> (관련 로그 및 스크린샷 생략)

```bash
# 관련 로그나 에러 메시지를 여기에 붙여넣습니다.
# API 요청 시 서버에서 401 Unauthorized 응답 발생
```

-----

### 2\. 원인 분석 (Analysis)

> 클라이언트가 **로컬 스토리지의 정적 플래그**에 인증 상태를 의존하여, **실제 서버의 토큰 만료 상태와 동기화되지 않는 것**이 핵심 원인입니다.

* **원인 1: [클라이언트 인증 상태 관리 방식의 문제]**

    * 클라이언트(프론트엔드)가 실제 토큰 유효성과 관계없이 \*\*로컬 스토리지의 단순 플래그 (`isLogin: true` 등)\*\*를 기준으로 로그인 상태를 관리했습니다. 이 플래그는 토큰 만료 시 자동으로 갱신되지 않아, \*\*가짜 로그인 상태(Stale Login State)\*\*가 발생했습니다.

* **원인 2: [인증 실패 응답 처리 부재]**

    * API 요청 시 서버가 401 Unauthorized 응답을 반환해도, 이를 **전역적으로 감지하고 로그아웃을 트리거하는 로직**이 부재했습니다. 이 때문에 새로고침 등의 강제적인 상황이 발생하기 전까지는 로그아웃 처리가 지연되었습니다.

-----

### 3\. 해결 방안 (Solution)

> **Axios 인터셉터 등을 활용**하여 401 에러를 전역적으로 감지하고, **로컬 스토리지의 인증 데이터를 즉시 초기화**하며 **강제로 로그인 페이지로 리다이렉트**하도록 처리했습니다.

* **[해결 방안 1]**: **Axios 인터셉터 및 전역 핸들러를 통한 401 감지 및 처리**
    * API 응답 코드 401(Unauthorized)을 전역적으로 감지하는 로직을 추가하여, 인증 실패가 발생하면 **지체 없이** 로그아웃 흐름을 시작하도록 했습니다.
* **[해결 방안 2]**: **인증 실패 시 로컬 스토리지 초기화**
    * 인증 실패 및 무한 재시도 감지 시, 클라이언트 측의 \*\*`localStorage`에 저장된 `auth` 항목(인증 상태 플래그 포함)\*\*을 즉시 제거하도록 처리했습니다. (커밋 메시지: "인증 실패 및 무한 재시도 감지 시 `localStorage`의 `auth` 항목 제거 처리 추가")
* **[해결 방안 3]**: **일관된 강제 리다이렉션 로직 적용**
    * 인증 실패 후, 사용자 경험을 최소화하기 위해 \*\*`window.location.replace`\*\*를 사용하여 뒤로 가기 히스토리를 남기지 않고 **강제로 로그인 페이지(`/`)로 리다이렉트**하도록 구현했습니다. (커밋 메시지: "인증 실패 후 강제로 로그인 페이지(`/`)로 리다이렉트 로직 구현")

-----

### 4\. 교훈 (Lessons Learned)

> 클라이언트의 인증 상태는 서버의 **실제 유효성 검사 결과**에 의존해야 하며, **전역적인 에러 핸들링**을 통해 상태 동기화의 일관성을 확보해야 합니다.

* **[교훈 1]**: **인증 상태 관리의 일관성 및 동기화**
    * 클라이언트의 로그인 상태는 로컬 스토리지의 정적 플래그가 아닌, \*\*서버와의 통신 결과(토큰 유효성)\*\*를 기반으로 관리해야 합니다. 인증 상태는 **항상 서버의 실제 상태와 동기화**되도록 설계해야 합니다.
* **[교훈 2]**: **인증 흐름의 캡슐화와 보호**
    * `LoadMeRoute`와 같이 인증 관련 흐름을 캡슐화하고 모든 라우트를 보호하는 구조를 도입하여, **인증 상태 보호를 강화**하고 인증 로직의 일관성을 향상시켜야 합니다. (커밋 메시지: "`LoadMeRoute`로 인증 관련 흐름 캡슐화 및 라우터 구조 단순화")
* **[교훈 3]**: **Axios 인터셉터 등을 이용한 전역 에러 핸들링의 중요성**
    * **401, 500 등 중요 API 응답 코드**에 대한 전역적인 인터셉터/핸들러 로직을 초기 설계 단계부터 적용하여, 인증 실패 또는 서버 오류 발생 시 **즉각적이고 일관된 사용자 피드백**을 제공해야 합니다.
